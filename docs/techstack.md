알겠습니다. YC 배치 스타트업 CTO의 관점에서, '신속한 MVP 개발 및 내부 베타테스트'라는 명확한 목표를 달성하기 위한 최종 기술 스택 추천 및 실행 계획입니다. 이 내용은 바로 복사하여 팀 내부 문서나 의사결정 자료로 활용하실 수 있도록 작성되었습니다.

### [프로젝트] 체험단 매칭 플랫폼 MVP 기술 스택 최종안

목표: 내부 베타테스트를 위한 첫 번째 프로토타입을 가장 빠른 속도로 완성한다.
핵심 가치: 신속한 개발, 오버엔지니어링 지양, 가장 쉬운 인프라.

### 최종 추천 기술 스택 요약

| 구분 | 기술 | 선정 이유 및 전략 |
|---|---|---|
| 백엔드 | Python Django | 압도적인 개발 속도, 강력한 내장 기능(Admin), 검증된 안정성과 확장성. |
| 프론트엔드 | Django Template (+ Bootstrap) | 별도 빌드 과정 없이 가장 빠르게 UI 구현. 오버엔지니어링 방지에 최적화. |
| 데이터베이스| SQLite (개발 및 배포) | (전략적 선택) 설정 제로(Zero-Config)로 인프라 고민을 제거, 개발 속도 극대화. |
| 배포 | Railway | GitHub 연동만으로 자동 배포. MVP 단계에서 가장 단순하고 빠른 인프라. |

---

### 1. 백엔드: Python Django

*   빠른 개발 속도: "Batteries-included" 철학에 따라 사용자 인증, 관리자 페이지(Admin), ORM 등 필수 기능이 내장되어 있습니다. 특히 '광고주용 체험단 관리' 페이지의 기본 기능은 Django Admin 커스터마이징만으로 신속한 프로토타이핑이 가능합니다.
*   안정성과 신뢰성: Django Software Foundation에 의해 활발히 유지보수되며, LTS(장기 지원) 릴리스를 통해 하위 호환성이 잘 보장됩니다. Breaking Change 걱정 없이 개발에만 집중할 수 있습니다.
*   미래 확장성: MVP 이후 API 서버가 필요할 경우, Django REST Framework(DRF)를 추가하여 손쉽게 확장할 수 있습니다. 이는 향후 고도화된 프론트엔드나 모바일 앱 도입을 위한 훌륭한 기반이 됩니다.

### 2. 프론트엔드: Django Template + Bootstrap

*   오버엔지니어링 방지: React, Vue 등 별도의 프론트엔드 프레임워크 도입은 MVP 단계에서 불필요한 복잡성을 야기합니다. Django 템플릿은 백엔드 로직과 화면을 가장 직관적으로 연결할 수 있습니다.
*   신속한 UI 개발: Bootstrap을 활용하면 미리 디자인된 UI 컴포넌트(버튼, 폼, 카드, 다이얼로그 등)를 즉시 가져와 사용할 수 있어, 디자이너 없이도 깔끔하고 반응형을 지원하는 UI를 빠르게 구성할 수 있습니다.

### 3. 데이터베이스: SQLite (개발 및 배포 통합)

이 결정은 일반적인 모범 사례는 아니지만, '내부 베타테스트'라는 명확한 범위와 '극도의 속도'를 최우선으로 하는 목표 하에 내리는 의도적이고 전략적인 선택입니다.

*   장점:
    *   설정 제로(Zero-Configuration): 별도의 데이터베이스 서버 설정, 연결 정보 관리, 네트워크 보안 등 인프라 관련 고민이 완전히 사라집니다. 데이터베이스는 프로젝트 내 단일 파일로 관리됩니다.
    *   궁극의 단순함: Railway 배포 시 다른 부가 서비스(DB Add-on) 없이 웹 서버 하나만으로 모든 것이 동작하여 잠재적 변수를 원천 차단합니다.
    *   속도 집중: 인프라에 쏟는 시간을 '0'으로 만들어, 100% 비즈니스 로직 구현에만 집중할 수 있습니다.

*   명확한 한계점 (인지 및 수용):
    *   동시성 문제: 쓰기(Write) 작업 시 파일 전체에 락(Lock)이 걸려 여러 사용자가 동시에 데이터를 수정하려 할 때 요청이 지연되거나 오류가 발생할 수 있습니다. (소수의 내부 테스터 환경에서는 문제가 될 확률이 낮습니다.)
    *   데이터 유실 위험: Railway와 같은 임시 파일 시스템(Ephemeral Filesystem) 환경에서는 재배포 시 데이터가 담긴 `db.sqlite3` 파일이 유실됩니다. 이는 아래의 액션 플랜으로 반드시 해결해야 합니다.
    *   확장 불가: 서버를 2대 이상으로 늘리는 수평 확장이 원천적으로 불가능합니다.

*   필수 액션 플랜 (데이터 유실 방지):
    Railway의 영구 볼륨(Persistent Volume) 기능을 사용하여 데이터베이스 파일이 재배포 시에도 유지되도록 설정해야 합니다.

    1.  Railway Volume 생성: Railway 프로젝트 대시보드에서 `New` > `Volume`을 선택하고, 마운트 경로(Mount Path)를 `/data`로 지정합니다.

    2.  `settings.py` 수정: 데이터베이스 파일 경로가 생성된 볼륨을 가리키도록 코드를 수정합니다.

        ```python
        # settings.py
        import os
        from pathlib import Path

        BASE_DIR = Path(__file__).resolve().parent.parent

        # Railway 볼륨 경로. '/data'는 Railway에서 설정한 Mount Path입니다.
        # 로컬 개발 환경을 위해 기본값(BASE_DIR)을 유지합니다.
        VOLUME_MOUNT_PATH = os.environ.get('RAILWAY_VOLUME_MOUNT_PATH', BASE_DIR)

        DATABASES = {
            'default': {
                'ENGINE': 'django.db.backends.sqlite3',
                # DB 파일이 영구 볼륨(/data/db.sqlite3)에 저장되도록 경로를 지정합니다.
                'NAME': os.path.join(VOLUME_MOUNT_PATH, 'db.sqlite3'),
            }
        }
        ```
    > 경고: 위 설정을 적용하지 않으면 재배포할 때마다 모든 데이터가 초기화됩니다.

### 4. 배포: Railway

*   Git Push 기반 자동 배포: GitHub 레포지토리를 연결하고 `git push`만 하면 빌드부터 배포까지 전 과정이 자동화됩니다. 배포에 쏟는 시간을 최소화하고 개발에 집중할 수 있습니다.
*   통합 인프라 관리: 웹 애플리케이션 서버, 환경 변수, 실시간 로그 확인 등 MVP 운영에 필요한 모든 기능을 직관적으로 제공합니다.
